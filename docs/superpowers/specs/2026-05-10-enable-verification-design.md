# 启用后确认与重试设计（2026-05-10）

## 背景

当前项目已经有“禁用后等待额度恢复再复查并重新启用”的机制，但缺少“启用请求发出后，确认 CPA 侧状态是否真的变为启用”的闭环。

现象是：
- 启用接口调用可能返回成功。
- 但 5 秒后再次读取账号详情时，账号仍可能保持 `disabled=true`。
- 现有逻辑会把第一次启用请求成功直接视为最终成功，导致：
  - `src/maintainer.py` 过早移除 tracked disabled account；
  - `enable_all_codex.py` 误报启用成功；
  - 实际上账号仍处于禁用状态。

本次设计目标是在不改动 CPA API 结构、不新增持久化格式的前提下，为两个启用途径补上“启用后确认”闭环。

## 目标

1. 在 `src/maintainer.py` 的自动重新启用流程中，只有确认账号实际已启用后，才视为启用成功。
2. 在 `enable_all_codex.py` 的批量启用流程中，只有确认账号实际已启用后，才视为启用成功。
3. 当启用后短时间内状态未落地时，自动补发启用请求并重试确认。
4. 最终失败时：
   - `src/maintainer.py` 保留“待下次处理”的语义；
   - `enable_all_codex.py` 记为失败但继续处理，并在末尾集中提示。
5. 尽量复用现有接口与状态结构，保持改动最小。

## 非目标

1. 不修改 `src/cpa_client.py` 的接口形态。
2. 不新增新的 JSON 状态文件或新的持久化 schema。
3. 不把批量脚本改造成读取 `.env` 配置的守护式工具。
4. 不引入后台异步确认队列或新的线程模型。

## 需求边界

### 适用入口

本次行为同时覆盖两个入口：

1. `src/maintainer.py`
   - 仅作用于“keeper 判定该账号已到可重新启用条件、准备从 disabled 恢复为 enabled”的自动重新启用路径。
2. `enable_all_codex.py`
   - 作用于批量启用脚本中，对 codex 账号逐个执行的启用操作。

### 重试规则

统一行为：
- 每次发起启用请求后，等待 5 秒。
- 5 秒后重新读取账号详情，确认 `disabled == False`。
- 如果仍未启用，则再次发起启用请求。
- 最多重试 3 次。

这里的“最多重试 3 次”按完整尝试轮次理解：
- 第 1 次：发起启用 → 等待 → 复查
- 第 2 次：补发启用 → 等待 → 复查
- 第 3 次：补发启用 → 等待 → 复查

任一轮复查确认 `disabled == False` 即立即成功结束。

### 失败后的收尾规则

#### `src/maintainer.py`

当 3 次尝试后仍未确认启用成功：
- 不移除 tracked disabled account；
- 不增加 `enabled` 统计；
- 记录最终失败日志；
- 保留为“待启用、等待下一次正常处理”的状态。

这意味着该账号仍会进入后续主巡检或现有 timer 复查路径，而不是被误判为已经恢复启用。

#### `enable_all_codex.py`

当 3 次尝试后仍未确认启用成功：
- 将该账号加入失败列表；
- 继续处理后续账号；
- 脚本最终返回非 0；
- 在最终汇总中明确打印需要人工检查的账号列表和“已达到最大重试次数仍未成功启用”的提示。

## 方案对比

### 方案 1：同步启用后确认循环（推荐）

做法：
- 在两个入口内都引入“启用请求 + 延迟确认 + 重试”的同步循环。
- 每次尝试使用现有接口：`set_disabled(..., False)` / `get_token_detail()`。

优点：
- 最直接对应问题根因。
- 对现有状态机侵入最小。
- 容易写测试和日志。
- 能同时覆盖自动流程与批量脚本。

缺点：
- 单个账号的启用路径会额外等待最多 15 秒。

### 方案 2：maintainer 用 timer 机制做短期确认，批量脚本同步确认

做法：
- maintainer 第一次启用后不阻塞等待，而是写入一个 5 秒后的短期复查计划。
- 批量脚本仍同步等待确认。

优点：
- maintainer 单次处理阻塞更少。

缺点：
- 语义更绕。
- 需要区分“额度恢复复查”和“启用确认复查”两类 timer 含义。
- 更容易引入状态歧义。

### 方案 3：只补一次读回确认，不做完整重试

做法：
- 启用后只等待 5 秒并复查一次。
- 若仍失败，仅记录失败，不继续补发启用。

优点：
- 实现最短。

缺点：
- 无法满足“最多重试 3 次”的需求。
- 对间歇性落地延迟的缓解不足。

### 推荐结论

采用方案 1：同步启用后确认循环。

原因：
- 它完整覆盖了“5 秒后复查、失败则补发启用、最多 3 次”的需求；
- 能保持 `maintainer` 与批量脚本的核心行为一致；
- 不需要重构现有 tracked timer 模型；
- 是满足目标的最小闭环改动。

## 详细设计

### 1. `src/maintainer.py`

#### 新增配置项

在 `src/settings.py` 中新增两个 settings，仅供 `maintainer` 使用：

- `CPA_ENABLE_VERIFY_DELAY_SECONDS`
  - 默认值：`5`
  - 含义：每次发起启用请求后，等待多少秒再复查账号状态。
- `CPA_ENABLE_VERIFY_MAX_ATTEMPTS`
  - 默认值：`3`
  - 含义：启用后最多进行多少轮“请求 + 等待 + 复查”。

校验规则：
- `CPA_ENABLE_VERIFY_DELAY_SECONDS >= 1`
- `CPA_ENABLE_VERIFY_MAX_ATTEMPTS >= 1`

#### 新增内部帮助方法

在 `src/maintainer.py` 中增加一个内部帮助方法，职责限定为：

- 接收账号名与 logger；
- 执行启用请求；
- 每轮等待配置的秒数；
- 重新读取账号详情确认 `disabled` 状态；
- 成功则返回成功；
- 否则继续下一轮；
- 达到最大次数后返回失败。

建议行为上只聚焦“启用确认”，不要顺手处理 tracked 状态移除、统计更新或额度逻辑。那些收尾动作保留在调用点，以避免职责混杂。

#### 调用点改造

当前自动重新启用路径在检测到：
- token 当前为 disabled；
- 且 tracked next check 到期；
- 且额度已低于阈值；

时，调用 `set_disabled_status(name, disabled=False, logger=logger)`。

这里改为：
- 调用新的“启用后确认”方法；
- 只有该方法确认成功后，才：
  - 移除 tracked account；
  - 记录“账号已重新启用”；
  - `enabled += 1`；
  - 将 `effective_disabled = False`。
- 如果最终失败，则：
  - 保持 tracked 记录不变；
  - 记录最终失败日志；
  - 保持 `effective_disabled = True`。

#### 日志设计

每轮至少记录：
- 第 N 次尝试启用；
- 启用请求是否调用成功；
- 等待多少秒后开始复查；
- 复查时读取到的账号状态；
- 是否进入下一次重试；
- 达到最大次数后的最终失败。

如果复查时 `get_token_detail()` 失败或拿不到 `disabled` 字段，也视为该轮未确认成功，并进入下一轮或最终失败。

### 2. `enable_all_codex.py`

#### 固定参数

脚本中直接写死：
- 延迟：5 秒
- 最大尝试次数：3 次

不引入 `.env` 配置，避免把一次性脚本复杂化。

#### 新增轻量帮助函数

在脚本中新增一个局部帮助函数，例如语义上等同于：
- `enable_account_with_verification(client, name, *, delay_seconds=5, max_attempts=3)`

职责：
- 对单个账号执行启用；
- 每轮等待 5 秒后重新读取详情；
- 复查成功则返回成功；
- 失败则重试；
- 达到上限后返回失败和失败原因。

该函数不负责遍历账号列表、不负责汇总，只服务单账号确认逻辑。

#### 主流程改造

`enable_accounts()` 中当前是：
- 遇到 disabled 账号，直接 `client.set_disabled(name, False)`；
- 返回值为真就记成功。

改为：
- 调用新的单账号确认函数；
- 只有确认通过后才记为成功启用；
- 最终失败则记入失败列表；
- 继续处理其他账号。

#### 汇总输出

脚本末尾在现有失败汇总之外，补充一条更明确的人工处理提示，例如语义上：
- 以下账号经过 3 次启用确认仍失败，请人工检查 CPA 侧状态。

这里不要求固定文案逐字一致，但必须满足：
- 明确说明已达到最大重试次数；
- 明确指出这些账号仍未成功启用；
- 明确提示人工检查。

## 数据与状态语义

### `maintainer` 的 tracked 状态

本次设计不改变现有 tracked JSON 结构。

当前 tracked disabled account 的语义是：
- 该账号由 keeper 纳入“后续再检查是否可恢复”的范围。

本次只强化一条规则：
- **只有确认账号真正启用后，才允许移除 tracked 记录。**

这能避免因为 CPA 状态延迟或未落地而把账号从 keeper 的后续处理链路中错误移除。

### 统计口径

`enabled` 统计只在“最终确认启用成功”后累加。

不再把“启用请求发送成功”视为统计成功。

## 错误处理

以下情况都视为单轮未确认成功：

1. 启用请求调用失败；
2. 启用请求调用成功，但 5 秒后查询仍是 `disabled=true`；
3. 5 秒后复查时获取账号详情失败；
4. 5 秒后复查时响应缺少可判定的 `disabled` 信息。

处理方式统一：
- 若还没到最大次数，则记录日志并进入下一轮；
- 若已到最大次数，则记录最终失败并返回失败结果。

## 测试设计

### `tests/test_settings.py`

新增测试覆盖：
- 读取 `CPA_ENABLE_VERIFY_DELAY_SECONDS`
- 读取 `CPA_ENABLE_VERIFY_MAX_ATTEMPTS`
- 默认值正确
- 非法值（0 或负值）被拒绝

### `tests/test_maintainer.py`

新增测试覆盖：
1. 第一次启用后复查仍禁用，第二次成功：
   - 最终成功；
   - tracked 记录被移除；
   - `enabled` 增加。
2. 连续 3 次后仍禁用：
   - tracked 记录保留；
   - `enabled` 不增加；
   - 日志包含最终失败信息。
3. 复查时 `get_token_detail()` 失败：
   - 该轮不算成功；
   - 若后续某轮成功，则整体成功。
4. 只有确认成功后才移除 tracked：
   - 在失败路径中仍能读到 tracked 状态。

测试应使用 `patch("src.maintainer.time.sleep")` 避免真实等待。

### `tests/test_enable_all_codex.py`

新增测试覆盖：
1. 第一次启用后复查即成功；
2. 第一次复查仍禁用，后续重试成功；
3. 3 次后仍失败：
   - 该账号记入失败列表；
   - 后续账号仍继续处理；
   - 脚本最终返回非 0；
   - 输出包含人工检查提示。
4. 复查读取失败后后续成功；

测试应 patch `time.sleep`，避免真实等待。

## 实施顺序

1. 在 `src/settings.py` 增加 `maintainer` 侧启用确认配置项及校验。
2. 在 `src/maintainer.py` 增加启用后确认帮助方法，并接入自动重新启用路径。
3. 在 `enable_all_codex.py` 增加脚本侧单账号启用确认帮助函数，并接入批量启用流程。
4. 补齐 `tests/test_settings.py`、`tests/test_maintainer.py`、`tests/test_enable_all_codex.py`。
5. 跑相关测试，确认自动流程和脚本流程的成功/失败语义一致。

## 风险与权衡

1. **处理耗时增加**
   - 单个账号的启用路径最多额外等待约 15 秒。
   - 这是换取状态确认闭环的直接代价。
2. **批量脚本总时长上升**
   - 如果失败账号较多，脚本整体运行时间会明显增加。
   - 但这比误报成功更可接受。
3. **CPA 状态读取不稳定时的误伤**
   - 如果 `get_token_detail()` 本身偶发失败，会消耗一次尝试次数。
   - 这仍符合“最多 3 次”的明确要求，比无限等待更可控。

## 验收标准

满足以下条件即可认为该设计完成：

1. `maintainer` 只有在复查确认账号已启用后，才移除 tracked 记录并累计 `enabled`。
2. `maintainer` 在 3 次内始终未确认启用成功时，账号仍保留待下次处理状态。
3. `enable_all_codex.py` 只有在复查确认账号已启用后，才记为成功。
4. `enable_all_codex.py` 对最终失败账号会继续处理后续账号，并在末尾打印人工检查提示。
5. 新增测试覆盖成功、重试成功、最终失败、复查读取失败等关键路径。
