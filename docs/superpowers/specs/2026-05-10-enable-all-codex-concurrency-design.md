# 2026-05-10 enable_all_codex 并发启用设计

## 背景

当前 [enable_all_codex.py](enable_all_codex.py) 按账号串行执行启用流程：发送启用请求、等待 5 秒、查询账号详情复查，必要时最多重试 3 次。面对数百个 codex 账号时，总耗时主要消耗在每个账号串行等待的 5 秒复查延迟上，批量启用速度过慢。

本次目标是仅改造批量启用脚本的执行模型，在不改变既有启用确认语义的前提下提升整体吞吐。`maintainer` 侧自动重新启用逻辑、配置体系以及 `CPAClient` 本身不在本次范围内。

## 目标

- 将 [enable_all_codex.py](enable_all_codex.py) 从串行账号处理改为并发账号处理。
- 保持单账号语义不变：`set_disabled(false)` → 等待 5 秒 → `get_auth_file()` 复查 → 最多尝试 3 次。
- 保持失败策略不变：单个账号失败时继续处理其他账号，全部结束后统一汇总失败账号并提示人工检查。
- 改善日志可读性：避免多个账号的日志在并发执行时交错。

## 非目标

- 不修改 [src/maintainer.py](src/maintainer.py) 的启用复查逻辑。
- 不修改 [src/cpa_client.py](src/cpa_client.py) 为异步 HTTP 客户端。
- 不引入 `asyncio` 或新的第三方依赖。
- 不把并发度做成 `.env` / settings 配置项。

## 方案选择

### 方案 A：线程池并发账号任务（采用）

使用 `ThreadPoolExecutor(max_workers=8)` 按账号并发执行完整启用流程。每个线程独立完成单账号的启用、等待、复查、重试和结果收集，主线程仅负责提交任务、收集结果、输出日志块和做最终汇总。

**优点**
- 保持 `CPAClient` 同步接口不变，改动面最小。
- 现有单账号流程和测试语义可直接复用。
- 对当前以网络请求和等待为主的脚本已足够显著提速。

**缺点**
- 不是真正的异步 I/O。
- 输出顺序会从“输入顺序”变成“任务完成顺序”。

### 方案 B：`asyncio` 调度 + 线程包装同步客户端（不采用）

外层改成 `asyncio`，实际请求仍通过线程执行同步 `CPAClient`。

**不采用原因**
- 复杂度高于线程池直用方案。
- 性能收益通常不会明显优于方案 A。
- 测试结构会更复杂，但对本次目标没有额外价值。

### 方案 C：`CPAClient` 全链路异步化（不采用）

将客户端和脚本一并改成真正异步 HTTP。

**不采用原因**
- 改动范围过大，回归风险高。
- 与“只优化批量启用脚本速度”的目标不匹配。

## 详细设计

### 并发模型

- 仅对“需要启用的 disabled codex 账号”提交并发任务。
- 并发上限固定为脚本常量 `DEFAULT_ENABLE_CONCURRENCY = 8`。
- 每个任务负责一个账号的完整生命周期：
  1. 校验账号字段
  2. 执行启用请求
  3. 等待 5 秒
  4. 查询详情复查
  5. 按需要重试，最多 3 次
  6. 产出结果与日志块
- 主线程使用 `as_completed()` 收集结果，因此日志输出和成功/失败统计按“完成顺序”产生，而不是输入顺序。

### 日志模型

当前脚本直接调用 `log()` 输出，会在并发场景中交错。改造后：

- 单账号任务内部不直接写全局 stdout。
- 每个任务维护自己的日志缓冲区，例如 `list[str]`。
- 单账号的所有日志都先写入缓冲区。
- 任务完成后返回完整日志块。
- 主线程在拿到任务结果后一次性输出该账号日志块。

这样可以保证：
- 不同账号之间不会出现日志穿插。
- 单个账号的“第几次尝试、等待、复查、成功/失败”会连续呈现。

### 输出顺序

- 账号日志块输出顺序按任务完成先后决定。
- 每个账号开头仍保留原有的 `[idx/total]` 信息，便于定位原始账号位置。
- 汇总信息继续在所有任务结束后统一输出。

### 结果模型

建议引入一个轻量结果对象（可用 `dataclass`、`NamedTuple` 或字典），至少包含：

- `name`
- `success`
- `already_enabled`
- `invalid`
- `failure_reason`
- `log_lines`

用途：
- 主线程只依赖结果对象做汇总，不关心单账号内部细节。
- 让日志输出和统计逻辑解耦，避免在线程之间共享可变统计状态。

### 函数边界

推荐把 [enable_all_codex.py](enable_all_codex.py) 调整为以下职责分层：

1. `enable_account_with_verification(...)`
   - 负责单账号的启用、等待、复查、重试。
   - 接受一个日志收集函数或缓冲容器，而不是直接打印。
   - 返回 `(success, failure_reason)` 或等价结果。

2. `process_account(...)`
   - 负责单个账号的外围处理：
     - 读取 `name/email/disabled`
     - 记录账号头部日志
     - 处理无效账号/已启用账号/待启用账号三种路径
     - 调用 `enable_account_with_verification(...)`
     - 产出单账号结果对象

3. `enable_accounts(...)`
   - 负责整体编排：
     - 拆分任务
     - 创建线程池
     - 提交待启用账号任务
     - 主线程逐个接收结果并打印日志块
     - 汇总成功、失败、已启用、无效统计
     - 输出失败账号与人工检查提示
     - 返回最终退出码

## 失败语义

以下行为保持不变：

- `name` 缺失仍视为失败并计入失败列表。
- `disabled is False` 仍直接跳过并计入已启用。
- 单账号启用确认失败后，不中断其他账号任务。
- 最终只要有任一账号失败，脚本退出码仍为 `1`。
- 最终仍输出：
  - 失败账号列表
  - “经过 3 次启用确认仍失败，请人工检查” 提示

## 测试策略

保留现有 [tests/test_enable_all_codex.py](tests/test_enable_all_codex.py) 的 `unittest` 风格，不引入异步测试框架。

### 保留验证

现有这些语义仍需覆盖：
- 已启用账号跳过
- 成功启用账号返回成功
- 启用后复查失败时会重试
- 详情复查失败时会重试
- 单个账号失败不影响其他账号
- 最终失败汇总与退出码正确

### 新增/调整验证

- 增加并发调度相关测试，验证 `enable_accounts()` 会通过线程池执行待启用账号。
- 增加日志块测试，验证某个账号的多行日志作为完整块输出，而不是与其他账号交叉。
- 为了避免真实线程调度带来不稳定：
  - 测试中优先 mock `ThreadPoolExecutor` 与 `as_completed()`，或
  - 让提交的任务在测试替身中同步执行，再返回类 future 对象。

这种方式可以稳定验证：
- 编排是否使用并发接口
- 汇总与日志语义是否保持正确
- 测试不会依赖真实线程切换时机

## 兼容性与风险

### 兼容性

- 外部使用方式不变，仍通过 `main()` 运行脚本。
- `CPAClient` 构造参数与接口不变。
- maintainer 流程完全不受影响。

### 主要风险

1. **CPA API 并发压力增加**  
   用固定并发 8 控制压力，避免一次性并发所有账号。

2. **日志时序变化**  
   输出顺序从输入顺序变为完成顺序，但通过保留 `[idx/total]` 和整块日志输出降低理解成本。

3. **测试脆弱性**  
   通过 mock 并发原语而不依赖真实线程调度，保持测试稳定。

## 成功标准

- 批量启用脚本在大量 disabled codex 账号下显著快于当前串行版本。
- 单账号启用确认语义与失败处理语义不变。
- 日志不会出现多账号交错污染。
- [tests/test_enable_all_codex.py](tests/test_enable_all_codex.py) 可稳定覆盖并发后的主要路径。
